# update-plan

プランモード終了直前にプランファイルの品質を確保し、設計書と同じ基準で合格して
から `ExitPlanMode` を呼ぶ統合スキル。4 フェーズで進める。

## Phase 1: コンテキスト収集

現在のプランファイルと、関連成果物を集める:
- 設計書 `docs/dev/spec.ja.md`
- プロジェクト文書 `README.md` / `README.ja.md` / `CLAUDE.md` / `AGENTS.md`
- マニフェスト `Cargo.toml` / `pyproject.toml`
- プランが触れる `src/` 配下のファイル

## Phase 2: 品質評価（100点満点）

`update-design` と同じ 5 カテゴリ × 20点で評価する:

| カテゴリ | 評価観点 |
| --- | --- |
| モジュール / struct 設計 | プランが Rust の境界を明確に扱っているか |
| 静的解析制約 | Python コード非実行の制約を崩さないか、変更スコープが妥当か |
| ルール / ポリシー | YOK ルールの判定ロジックと既定挙動への影響 |
| エラー処理 | Result 伝播、unwrap 回避、終了コード規約の遵守 |
| テスト容易性 | カバレッジ維持戦略と具体的なテスト計画 |

合格には 90点以上が必要。

## Phase 3: 整合性チェック

- プランと既存コードベースの整合（シンボル名・パス）
- 現行設計書との突き合わせ
- 手順の論理順序が依存関係を尊重しているか

## Phase 4: 改善と確定

- 検出した課題を P0 / P1 / P2 で分類しプランへ反映する
- 採点と主要所見の検証サマリをプランに付記する
- その後 `ExitPlanMode` を呼ぶ
